## 一、项目背景

### 1.1 DataMatrix 是什么

DataMatrix 是美团内部的**云搜数据平台**，核心架构是：

```
业务数据源（MySQL/Blade/Hive/Pigeon/MThrift）
    ↓ 数据集成（数据镜像构建）
缓存层（数据镜像 = ODS 层）
    ↓ 屏蔽异构性
数据建模层（异构查询 → 统一索引）
    ↓
搜索应用（ES8 索引）
```

引入中间镜像层的核心动机：
1. 减少建模层对业务数据源的查询压力
2. 屏蔽上游异构性，让建模层专注自身逻辑

## 二、选型分析

### 2.1 三种形式对比

#### 形式 A：本地文件 + Git（Markdown/Obsidian）

| 维度 | 评分 | 详细说明 |
|------|------|---------|
| 维护成本 | ★★★★☆ | 纯文本无迁移风险；git diff 自然追踪变更；离线可用 |
| 检索质量 | ★★★☆☆ | 关键词搜索够用；跨服务语义检索弱，需额外工具 |
| AI 友好性 | ★★★★★ | OpenCode/Claude 直接读取本地文件，零 API 开销、零延迟 |
| 团队协作 | ★★☆☆☆ | 个人维护极优；多人协作需约定规范，摩擦较高 |
| 长期稳定 | ★★★★★ | 无供应商锁定；Markdown 格式永久可读 |

**核心优势**：AI agent 的天然工作介质。Claude 读取本地 `.md` 文件的速度和成本远优于通过 citadel CLI 访问学城。

**核心劣势**：搜索能力依赖文件结构和命名，规模大了之后需要加索引层。

---

#### 形式 B：美团学城（现有体系）

| 维度 | 评分 | 详细说明 |
|------|------|---------|
| 维护成本 | ★★★☆☆ | 已有大量内容，但 AI 读取需 citadel CLI 中转，有认证延迟 |
| 检索质量 | ★★★★☆ | 结构化导航强；内置搜索；但 AI 语义检索弱 |
| AI 友好性 | ★★☆☆☆ | 需要 `oa-skills citadel getMarkdown` 中转；每次认证 |
| 团队协作 | ★★★★★ | 天然支持多人协作、评论、@mention、版本历史 |
| 长期稳定 | ★★★☆☆ | 依赖美团内部平台；存在迁移风险 |

**核心优势**：团队协作、正式设计文档的权威来源，已有大量沉淀。  
**核心劣势**：AI 每次访问都要走 CIBA 认证，不适合作为 AI agent 的实时工作介质。

---

#### 形式 C：向量数据库（RAG）

| 维度 | 评分 | 详细说明 |
|------|------|---------|
| 维护成本 | ★★☆☆☆ | 需要 embedding pipeline、索引维护、模型托管成本 |
| 检索质量 | ★★★★★ | 语义搜索最强；支持 hybrid search（BM25 + embedding） |
| AI 友好性 | ★★★★★ | 专为 RAG 设计；agent 检索效果最优 |
| 团队协作 | ★★★☆☆ | 需要额外 UI；不适合人类直接阅读和编辑 |
| 长期稳定 | ★★★☆☆ | 依赖向量数据库服务（Weaviate/pgvector）；索引需持续维护 |

**核心优势**：大规模语义检索、跨服务问答、代码理解 RAG。  
**核心劣势**：**当前阶段成本不合算**。DataMatrix 文档量约 30-50 篇，关键词搜索完全够用；向量化的边际收益不足以覆盖基础设施成本。

---

### 2.2 关键决策因素

**你是主要维护者（甚至唯一维护者）**

这是最重要的约束。多人协作场景下学城的优势（评论、@mention、权限管理）在个人维护场景下几乎不体现。而 AI agent 的工作效率，本地文件远高于学城。

**学城已有大量内容，不应重复建设**

学城现有 12 个模块、数十篇设计文档，是权威来源。知识库建设的工作不是"把学城内容搬到另一个地方"，而是"对学城内容做 AI 可读的摘要和索引"。

**向量数据库的引入时机**

向量数据库的价值在于：文档量大（>100 篇）、需要跨文档语义检索、需要自动问答。当前阶段这三个条件都不完全满足，引入会增加维护负担而收益有限。合理的触发条件是：KB 文档量超过 80 篇，或者出现"我知道某个知识点存在，但找不到在哪个文件"的情况。

---

### 2.3 推荐：分层混合架构

```
┌─────────────────────────────────────────────────────┐
│  Layer 1: 学城（权威 & 协作层）                      │
│  职责：正式设计文档、需求文档、架构决策、团队共享     │
│  访问方式：浏览器 / citadel CLI                      │
│  维护者：团队（含华雨）                              │
└──────────────────────┬──────────────────────────────┘
                       │ 摘要归纳（单向，学城→本地）
┌──────────────────────▼──────────────────────────────┐
│  Layer 2: 本地 Markdown + Git（AI 工作层）           │
│  职责：AI 可读摘要、代码理解、踩坑日志、决策记录     │
│  访问方式：OpenCode/Claude 直接读取                  │
│  维护者：华雨（个人维护）                            │
└──────────────────────┬──────────────────────────────┘
                       │ 未来按需引入（触发条件：文档>80篇）
┌──────────────────────▼──────────────────────────────┐
│  Layer 3: 向量检索（可选，暂缓）                     │
│  职责：大规模语义搜索、跨服务问答                    │
│  引入时机：文档量 >80 篇 或 出现检索瓶颈             │
└─────────────────────────────────────────────────────┘
```

**两层分工的核心原则**：
- 学城是「写给人看」的权威文档
- 本地 KB 是「写给 AI 看」的工作介质
- 两者不重复，本地 KB 不替代学城，只做摘要和补充

## 五、成本评估

### 5.1 初始建设成本

| 阶段 | 工作量 | 说明 |
|------|--------|------|
| 目录结构初始化 | 30 分钟 | 建目录、git init、写 AGENTS.md |

- **技术选型**：pgvector（轻量）或 Weaviate（功能完整）
- **预估成本**：1-2 天初始化 + 每月少量维护
